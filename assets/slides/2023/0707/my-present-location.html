SaaS系の共通項


  • 「使い捨て」感がある（一度実行したらそれっきり）
    • 毎回起動に時間かかる
  • ローカルとは違う
    • 検証が難しい
  • お金をかけないとなかなかパワーが上がらない
    • スペック不足だとプラットフォームごと移すことになる
  • 設定(yaml)をプログラミング「させられている」
    • インデントの制約地獄だし間違いに気付きにくい
  • Orb とか Actions はほしい
    • インストール大事だけど大変

Jenkinsを選択した理由


  • 使い捨て「たい」時に捨てたいところを捨てられる
    • docker rmi, …
  • ローカルと同じ操作性で利用可能
    • 差はスペック
  • セルフホストである程度費用を抑えられる
    • リザーブドすればさらに
  • jenkinsfile や groovy で実装
    • 条件分岐もわかりやすく書ける
  • インストール大作戦は某Bitbucketでだいぶ慣れた
    • docker職人養成所感

EC2(サーバー)の使い所


  • docker・compose を使える（ECS では使えない）
    • たぶん、ローカルでやるのとかなり近い
  • EFS をマウントすれば空き容量の心配をそんなにしなくて良い
    • とはいえ正しく「使い終わったら削除」すべき
  • 再構築を容易に可能としておけば最強のCIサービスになり得る
    • 「前提を揃える」を使いながらにして実践できる

マルチアーキテクチャ


  • docker build -> ecr push 時に amd64 のも arm64 のもやる
    • manifest を作る必要があることを知った
    • cypress で使うライブラリに
      arm64 対応してないものがあることを知った
    • M1 Mac 環境で amd64 の image を build する
      のにはだいぶ時間かかることを知った

まだ、過渡期なのだ。
知れることが多いのはありがたく、
きっと将来に活きる

マルチアカウント


  • セキュリティでよく言われている「マルチアカウント」
  • 周りを見回しても、まだそんなには普及していないようだ
  • とはいえ、今後は意識して、問題の影響を小さくしていきたい
  • マイクロサービス化の意義は、そういうところにありそう
